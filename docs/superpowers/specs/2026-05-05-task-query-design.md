# 任务配置与查询一致性设计

## 目标

优先保障高德地图与天地图链路，使任务配置、界面显示、运行时拆分、分页、调度、重试在同一套语义下工作，并保证配置文件与界面显示一致。

## 关键约束

- 资源在配置中优先保存唯一编码，界面按编码回显。
- 行政区域采用“保留用户原始选择，运行时再展开”的模型。
- 行政区域缓存统一使用高德标准命名，仅在用户主动更新行政区或运行时缓存缺失且明确需要补齐时才写入 region_cache.json。
- 查询默认串行执行：并发数为 1，子任务间隔使用配置中的秒级间隔。
- 每次请求失败最多重试 2 次，即总尝试次数最多 3 次。

## 设计

### 1. 任务配置与回显

- 高德/天地图任务资源保存为 data_type_tree 对应叶子编码。
- 加载任务时，界面根据编码遍历 data_type_tree，找到对应叶子并回勾。
- 行政区域配置只保存用户原始选择的 admin_regions，不在保存时展开“全部”。
- 加载任务时直接按 admin_regions 原始值恢复树勾选，保证配置文件与界面一致。

### 2. 运行时查询

- run_task 基于 admin_regions 原始选择动态拆分子任务。
- 省级“全部市”按 region_cache.json 展开为多个城市子任务。
- 市级“全部区县”按 region_cache.json 展开为多个区县子任务；若缓存缺失，按需调用高德补齐并写回缓存。
- 高德/天地图查询时使用编码作为资源类型；分页由 providers.py 累计所有页结果并回传。
- 运行结束后继续按现有去重逻辑汇总最终结果。

### 3. 高级配置

- max_concurrency 在运行层和 GUI 层都收口为 1。
- province_expand_delay_seconds 作为所有子任务请求的最小间隔。
- 不再走并发子任务执行分支。

### 4. 自动查询

- 软件开启后，调度线程按 check_interval_minutes 轮询。
- 每个任务根据最近一次 success 日志的 run_time 计算是否到期。
- 仅当当前时间减去最近成功时间大于 interval_days 时自动执行。

### 5. 重试

- 单个子任务请求失败后立即重试，最多额外重试 2 次。
- 重试发生在 execute_subtask/fetch_with_delay 链路，确保分页与子任务统计一致。
- 最终失败需写入日志并反馈到界面。

## 测试策略

- 回归测试 1：任务保存/加载时，资源编码与 admin_regions 原始选择保持一致。
- 回归测试 2：行政区域运行时展开为多个子任务，并正确累计分页结果。
- 回归测试 3：调度只根据最近成功日志判断到期。
- 回归测试 4：单次请求失败会再尝试 2 次，第三次成功或最终失败都能被正确记录。

## 范围

本轮优先覆盖高德与天地图。百度链路只做不破坏现状的保护，不作为本轮验收主目标。
